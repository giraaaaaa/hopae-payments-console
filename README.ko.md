# Hopae Software Engineer — 과제

## 1. 배경

당신은 결제 플랫폼의 어드민 대시보드를 만드는 엔지니어입니다. 가맹점이 자사의
결제 내역을 확인하는 콘솔 화면 일부를 구현합니다.

## 2. 과제의 구성

이 과제는 백엔드 역할을 할 **mock 서버**와, 결과물의 레이아웃을 담은 **와이어프레임**을
제공합니다.

### mock 서버 (백엔드)

- **제공**: 데이터 스키마(`mock-server/db.json`), **인증(auth) API**, 그리고 데이터를
  시간에 따라 변화시키는 **백그라운드 엔진**. (TypeScript로 작성되어 있고, 빌드 없이
  `tsx`로 실행됩니다.)
- 이 과제는 의도적으로 **백엔드 API를 완성해서 주지 않습니다.** 요구사항을 구현하는 데
  사용할 트랜잭션 **목록 / 환불(refund) API는 라우트·DTO 설계부터 구현까지 모두
  직접** 작업해야 합니다. (`routes/transactions.ts`에 `501` stub으로 표시되어 있습니다.)
- 즉 이 과제는 "주어진 API를 소비하는 프론트엔드"가 아니라, **API 표면부터 직접 설계하고
  그 위에 프론트엔드를 얹는** 과제입니다.
- **제공 파일(읽기만 해도 됩니다)**: `lib/auth.ts`, `lib/store.ts`, `server.ts`,
  `types.ts`. **당신이 만드는 파일**: `routes/transactions.ts` — 파일을 열면
  **`START YOUR CODE HERE`** 마커가 있습니다. 자세한 내용은 `mock-server/README.md`를
  참고하세요.
- 단, "제공 파일은 건드리지 마세요"가 아닙니다 — 제공 파일을 포함해 **더 나은 구현을
  위해 필요하다고 판단되면 어떤 파일이든 자유롭게 수정·확장**해도 됩니다. (무엇을 왜
  바꿨는지는 README에 남겨주세요.)

### 와이어프레임

와이어프레임(`wireframes.html`)은 **전체적인 틀을 보여주는 참고 자료일 뿐, 디자인
가이드가 아닙니다.** 아래 요구사항만 충족한다면, 그 위의 표현 방식은 자유입니다:

- 와이어프레임에 그려진 UI라도 **핵심 플로우가 아니라고 판단되면 구현에서 빼도** 됩니다.
- 와이어프레임이 제안하는 UI/UX보다 더 나은 방식이 있다면 **본인이 더 좋다고
  생각하는 대로 바꿔** 구현해도 됩니다.
- 와이어프레임에 **없는 UI라도, 더 나은 사용자 경험을 위해 필요하다고 판단되면 얼마든지
  추가**해도 됩니다.

요컨대 와이어프레임은 "이렇게 생겨야 한다"가 아니라 "대략 이런 화면들이 있다" 정도로
받아들이고, **더 나은 제품을 만드는 판단**을 보여주세요. 단, 아래 요구사항 자체는
충족해야 합니다.

## 3. 구현 요구사항

**기술 스택**: **React + TypeScript** (필요하다면 Next.js 등 React 기반 프레임워크를
사용해도 좋습니다). 번들러·라우팅 구성과 상태 관리·데이터 페칭·스타일링·UI 키트
(예: shadcn) 등 그 외 라이브러리도 자유입니다. (선택 이유는 README에 — §4 참고.)

### 로그인

- 이메일 + 비밀번호 폼으로 `POST /api/auth/login` 호출
- 성공 시 토큰 저장 후 대시보드로 이동, 실패 시 사용자에게 명확한 피드백
- 테스트 계정: `demo@hopae.com / password123`

### Sandbox / Production 환경 전환

- Sandbox ↔ Production 환경 스위처를 노출하고, 전환 시 데이터가 그에 맞게 갱신됩니다.
- **Sandbox**는 테스트 환경(실제 돈이 움직이지 않음), **Production**은 실제 운영
  환경(진짜 매출·고객·환불)입니다.
- 두 환경의 차이를 잘 염두에 두고, 그에 기반해 UI/UX를 설계해주세요. 어디까지 어떻게
  차등을 둘지는 본인 판단이며, **그 결정과 이유를 README에 적어주세요.**

### Transaction List

- 트랜잭션 목록 (ID, 금액, 통화, 상태, 고객명, 생성일)
- **준실시간 갱신**: mock 서버가 새 트랜잭션을 자동으로 생성하고 기존 트랜잭션의 상태를
  바꿉니다(`pending` → `succeeded`/`failed`). 이 변화가 사용자의 흐름을 방해하지 않으면서
  합리적인 방식으로 어드민 콘솔 UI에 반영되어야 합니다.
  *어느 시점에 어떤 방식으로 목록을 갱신할지는 전적으로 본인 선택입니다.* 내가 이 콘솔의 사용자라는 관점에서 최대한 자연스러운 UX를 고민해 주세요.
- **상태 필터**: 상태별(`succeeded`/`pending`/`failed`/`refunded`)로 좁혀 보기
- **검색**: 트랜잭션 ID와 고객 이메일 기준 검색
- 필터·검색을 어디에서 어떻게 처리할지는 본인 선택이며, **그 판단의
  근거를 README에 적어주세요.**
- 목록 api는 pagination 방식으로 구현해주세요. 프론트에서 어떻게 api를 호출해서 어떻게 렌더링할지도 본인의 판단입니다.
- **환불(Refund)**: 목록에서 트랜잭션을 환불할 수 있어야 합니다(예: 행 단위 액션 +
  확인 단계). 환불을 데이터·목록·API에서 **어떻게 모델링할지는 본인 설계**이며, 환불
  전후 목록에 어떻게 반영할지도 함께 고려해주세요. 환불 도중 해당
  행이 백그라운드로 갱신되는 경우의 처리도 본인 판단입니다.

### README를 실제 구현만큼 중요하게 봅니다

- **기술 선택과 그 이유** — 어떤 라이브러리·접근을 왜 골랐는지. 특히 **환경(env) 상태를
  어디에 저장하는지**와 **준실시간 갱신을 어떻게 구현했는지**는 명시해주세요.
- **내린 의사결정** — 이 과제는 의도적으로 많은 의사결정 포인트를 열어 두었고, 시간 제한
  take-home의 특성상 명세에 없는 결정이 더 많이 필요합니다. **어떤 부분에서 어떤 결정을
  왜 내렸는지** 구체적으로 적어주세요. (예: 설계한 API·DTO의 형태와 근거, 환경 전달 방식,
  환불 모델링, 새 트랜잭션을 UI에 드러내는 방식, 그리고 시간이 더 있었다면
  개선하고 싶은 점이나 의도적으로 단순화·생략한 부분 등.)
- 모든 화면을 완성하지 못해도 괜찮습니다. **못 끝낸 부분도 무엇을 고민했고 어떻게
  접근하려 했는지를 README에 녹여주세요.**

### 전반적인 사용성
- 로딩 / 에러 / 빈 상태 / 성공 / 갱신 등 각 상태를 사용자에게 어떻게 알릴지 섬세한 고민을 해주시면 좋습니다.
- 비정상적인 접근이나 예외 상황에 대한 가딩 및 적절한 피드백이 대응되어 있으면 좋습니다.
- 여기서 말하는 사용성은 **시각적 완성도(예쁜 디자인)가 아닙니다.** 디자인이 다소
  투박하더라도, **실제 고객 경험을 어떻게 설계했는지 — 사용자가 이 콘솔과 어떤 방식으로
  상호작용하게 만들었는지 — 가 훨씬 중요합니다.**
